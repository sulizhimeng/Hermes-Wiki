## 1. 架构级新增模块

### 1.1 多 Agent Kanban 板（v0.13.0）

`hermes_cli/kanban*.py` + `tools/kanban_tools.py` 引入了 **真正的多 Agent 协作底层**：一块 SQLite 看板（`<HERMES_HOME>/kanban.db`），多个 worker 进程从板上认领任务，心跳续期，死锁回收，重试预算。这是对原有 `delegate_task`（单进程同步 fork）的**异步分布式替身**。

源码事实：

- 板存储：`hermes_cli/kanban_db.py:809` 起声明 5 张表 — `tasks` / `task_links` / `task_comments` / `task_events` / `task_runs` / `kanban_notify_subs`。
- 板隔离：默认板 `<hermes_root>/kanban.db`，额外板放在 `<hermes_root>/kanban/boards/<slug>/`，**profile 之间共享同一根板**（profile 不是隔离边界，板才是）。
- Swarm 拓扑（`kanban_swarm.py`）：在已有 task 图上插入 `planning root → 多个并行 specialist → verifier → synthesizer` 子图，黑板用 root task 的 JSON `task_comments`，**不引入第二个 scheduler**。
- 工具表面（`tools/kanban_tools.py`）：9 个工具 `kanban_show / list / complete / block / heartbeat / comment / create / unblock / link`，仅在 `HERMES_KANBAN_TASK` 环境变量已设置（dispatcher 派生 worker 时）或配置启用 orchestrator profile 时注册到 schema —— 普通 `hermes chat` 看不到这些工具。
- 诊断（`kanban_diagnostics.py`）：1058 行规则引擎，发出可恢复的 `Diagnostic`（kind / severity / 建议动作）— 幻觉 ID、spawn 崩溃循环、超时 blocked 等。规则**只读**，从 (task, events, runs) 推导，dashboard 和 `hermes kanban diagnostics` 共用。
- 自动分解（`kanban_decompose.py`）：`hermes kanban decompose` 用辅助 LLM 把一个 triage 任务拆成子图，按 profile roster 分派 assignee；`kanban specify` 是其退化形式（不拆只 tighten）。
- CLI 入口：`hermes kanban` 子命令族 + `/kanban` 斜杠命令。任何 backend（Docker / Modal / SSH 等）的 worker 都通过**工具调用**而不是 shell `hermes kanban` 来更新板，因为容器内通常没安装 hermes 也访问不到宿主的 `kanban.db`。

详见新页面 [[multi-agent-kanban]] 和补丁后的 [[multi-agent-architecture]]。

### 1.2 `/goal` Ralph 循环（v0.13.0）

`hermes_cli/goals.py`（762 行）实现"持续目标"：用户给出目标后，每回合结束时让辅助模型当 judge 评估是否完成，未完成则把 continuation prompt 当作 user message 喂回去，直到完成 / 用完轮次 / 用户手动暂停 / 用户发新消息。

源码事实：

- judge 失败 **fail-OPEN**（返回 `continue`），靠 turn budget 兜底。
- 用户消息**抢占** continuation prompt，并暂停 goal loop 那一轮（仍然 re-judge）。
- 状态用 SessionDB `state_meta` 表，键为 `goal:<session_id>`，`/resume` 自动拾取。
- 不修改 system prompt 也不切 toolset —— **prompt cache 不失效**。

类似的"代际原语"还有：

- `/steer <prompt>`（v0.11.0/v0.13.0 强化）—— 中途插入提示，agent 在下一次 tool call 之后看到，不打断本回合；ACP 客户端（Zed / VS Code / JetBrains）也支持。
- `/queue` —— 把后续消息排队，等当前回合结束后投递。
- `/handoff` (v0.14.0) —— **现场迁移整个 session**（含 messages / tool calls / context）到目标模型、persona 或 profile，不丢上下文。

详见新页面 [[goal-loop-and-steering]]。

### 1.3 ProviderProfile 插件系统（v0.13.0）

`providers/base.py:39` 定义了 `@dataclass class ProviderProfile`，声明性描述一个推理 provider 的所有"差异点"（auth / endpoints / 客户端怪癖 / 请求级怪癖）。`providers/__init__.py` 实现了**延迟扫描** + **用户覆盖**机制：

- 内置插件位于 `plugins/model-providers/<name>/`（29 个：anthropic / openrouter / nous / openai-codex / gemini / xai / nvidia / arcee / stepfun / ai-gateway / gmi / azure-foundry / minimax / bedrock / huggingface / kilocode / kimi-coding / ollama-cloud / opencode-zen / qwen-oauth / zai / alibaba / alibaba-coding-plan / copilot / copilot-acp / custom / deepseek / novita / xiaomi）。
- 用户插件位于 `$HERMES_HOME/plugins/model-providers/<name>/`，**同名时覆盖内置**（last-writer-wins）。
- 向后兼容：`providers/*.py` 单文件 profile 仍被 `pkgutil.iter_modules` 发现。

每个 plugin 是一个目录，含 `__init__.py`（导入时调用 `register_provider(profile)`) 和 `plugin.yaml`（manifest: name / kind / version / description）。

这把 [[provider-transport-architecture]] 里的"transport 处理数据路径"和"profile 描述 provider 差异"彻底分开了：transport（4 个：anthropic / chat_completions / bedrock + codex 系列）少而稳，profile 多且常增减。

详见新页面 [[provider-plugin-system]] 和补丁后的 [[provider-transport-architecture]]。

### 1.4 `hermes proxy` —— OAuth 提供方的 OpenAI 兼容本地代理（v0.14.0）

`hermes_cli/proxy/`（server.py 308 行 + cli.py 142 行 + adapters/）启动一个 `http://localhost:<port>/v1/...` HTTP 服务器，把请求**仅替换 Authorization 头**后透传给上游。上游可选：

- `nous`：Nous Portal（解析 `~/.hermes/auth.json` 的 OAuth state，按需刷新 token）
- `xai`：xAI Grok via SuperGrok OAuth（用 `CredentialPool` 拉取最新 access token）

允许的路径白名单写死在每个 adapter（如 xai：`/responses` `/chat/completions` `/completions` `/embeddings` `/models`），其他路径返回 404，防止流量泄漏。

效果：Codex CLI / Aider / Cline / Continue / 任意 OpenAI 客户端可以用 Claude Pro / ChatGPT Pro / SuperGrok 订阅作为后端，**不需要 API key**。

详见新页面 [[hermes-proxy]]。

### 1.5 Checkpoints v2（v0.13.0）

`tools/checkpoint_manager.py` 头部注释明确"v2"，是对状态持久化的**重写**：

- 单一共享 git store：`~/.hermes/checkpoints/store/`，所有项目共用一个 bare-ish 仓库，object DB 跨 worktree 去重。
- 索引按项目分：`refs/hermes/<hash16>`（per-project ref）+ `indexes/<hash16>`（per-project index）+ `projects/<hash16>.json`（meta）。
- 自动维护：`prune_checkpoints` 删孤儿 ref + 过期 ref（按 `retention_days`），跑 `git gc --prune=now`，size-cap 按 `max_total_size_mb` 截断。
- 旧式（pre-v2）的 per-project shadow 仓库会在第一次启动时自动归档到 `legacy-<timestamp>/`。

老的 per-project shadow 仓库会重复存储相同 blob —— v2 一次性解决磁盘膨胀（原文 ~40MB/worktree × N → 单仓库去重）。

### 1.6 LSP 语义诊断（v0.14.0）

`agent/lsp/`（11 个文件）把真正的语言服务器（pyright / gopls / rust-analyzer / typescript-language-server 等）作为子进程跑起来，把 `textDocument/publishDiagnostics` 接到 `write_file` / `patch` 的 **post-write lint delta** 里。

源码事实：

- **门控在 git 工作区检测**：cwd 在 git repo 内才启动 LSP daemon，否则退回 `file_operations` 原有的进程内语法检查。避免在 home 目录（gateway 聊天）启 daemon。
- 公共 API：`agent.lsp.get_service()` → `svc.touch_file(path)` / `svc.diagnostics_for(path)`，调用点在 `tools.file_operations.FileOperations._check_lint_delta`。
- 这是对 v0.13.0 基础 lint（Python / JSON / YAML / TOML 纯语法）的**真升级**：现在是类型错误、未定义符号、缺失 import 等语义级。

详见新页面 [[lsp-integration]]。

### 1.7 i18n / 国际化（v0.13.0）

`agent/i18n.py` + `locales/` 引入了**薄切片**国际化：仅翻译"hermes 自己显示给用户的静态字符串"（审批 prompt、几条 gateway 斜杠命令回复、restart drain 通知）。**Agent 生成的文本、日志行、错误 traceback、工具输出、斜杠命令描述全部保留英文**。

源码事实：

- 解析顺序：`lang=` 参数 > `HERMES_LANGUAGE` 环境变量 > `display.language` 配置 > `"en"`。
- 目录 `locales/`：af / de / en / es / fr / ga / hu / it / ja / ko / pt / ru / tr / uk / zh / zh-hant — **16 个 YAML 文件**（release notes 强调的"7 个生产语言"是有 first-class 支持的子集：zh / ja / de / es / fr / uk / tr）。
- 文档站（`website/`）同步增加了 zh-Hans。

详见新页面 [[i18n-and-locales]]。

## 4. 网关 / 会话改动

### 4.1 状态持久化向 SQLite 统一（v0.13.0 + post-v0.14.0）

历史上 gateway 的会话/消息有两条存储路径（SQLite `state.db` + JSONL 文件）。post-v0.14.0 期一系列 commit 砍掉了 JSONL 路径：

```
33a3cf532 docs(sessions): state.db is canonical for gateway messages
b4b118c20 refactor(gateway): drop _append_to_jsonl from mirror
351fdcc6e refactor(gateway): stop writing JSONL in append_to_transcript / rewrite_transcript
971cfaa38 refactor(yuanbao): migrate recall to load_transcript()
024a8e3ee refactor(gateway): drop JSONL fallback in load_transcript
1d27be0ff test(gateway): pin SQLite-only load_transcript behaviour
9d793e8e5 docs(session-log): state.db is canonical; ~/.hermes/sessions/ is legacy
ce2678518 refactor(session-log): delete _save_session_log and all callers
eeb747de2 feat(sessions): opt-in per-session JSON snapshot writer
```

含义：

- 网关消息 transcript 的**权威**来源是 `state.db`。
- `~/.hermes/sessions/*.json` 现在是 legacy，**不再由 hermes 写入**；如需 JSON 快照，启用 opt-in `per-session JSON snapshot writer`。
- platform_message_id 现在在 `state.db` 持久化（commit 31a0100），yuanbao 等平台可精确按消息 ID 召回。

[[session-search-and-sessiondb]] / [[gateway-session-management]] 已对 SQLite 路径有覆盖；本节明确"JSONL 已退役"。

### 4.2 Gateway 重启自动续约（v0.13.0）

`gateway/run.py:3514-3565` 在 startup 期扫描"上次跑到一半被进程终止"的 session，标记 `resume_pending`，等下一个 user message 到来时**从既有 conversation 续上**而不是新建 session。这就是 release notes 里的"sessions survive restarts"。

### 4.3 API Server: `X-Hermes-Session-Key` 头（v0.13.0）

`gateway/platforms/api_server.py:780`：调用方在 `/v1/chat/completions` 或 `/v1/responses` 里附带 `X-Hermes-Session-Key: <opaque>` 头，hermes 把这个 key 透传给 memory provider 当 stable session 标识。**只在已配置 API key 鉴权时才接受**（line 802 显式拒绝未配置鉴权的调用）。意味着远程 API 用户也能拿到 per-session 长期记忆。

### 4.4 Webhook 直送（direct-delivery，v0.11.0）

`gateway/platforms/webhook.py:485-518`：webhook 路由可标记 `deliver_only=true`，**完全绕过 agent**，把 payload 直接投递到指定平台/频道。零 LLM 推送，用于 alerting / uptime / 事件流。日志关键字 `[webhook] direct-deliver`。

### 4.5 Cron `no_agent` 模式（v0.13.0）

cron job 可以跳过 agent，只跑一个脚本。空 stdout 安静；非空 stdout 逐字投递到目标平台。详见 [[cron-scheduling]]。

### 4.6 平台 allowlist 全覆盖（v0.13.0）

`allowed_channels` / `allowed_chats` / `allowed_rooms` 等配置在 Slack / Telegram / Mattermost / Matrix / DingTalk 全部接通。配合 v0.13.0 的安全潮（见第 7 节）一起。
