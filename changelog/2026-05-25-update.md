## 1. Docker `s6-overlay` 接管 PID 1 — BREAKING

`e0e9c895d feat(docker)!: replace tini with s6-overlay as PID 1`。这是自 v0.14.0 以来对容器启动入口**首次破坏式**重构。范围横跨 21 个 commit（`d36461d80` 提案文档 → `4f416fc40` 非特权 hermes 用户 lifecycle 收尾），合计 +~3200 行代码 + 测试。

### 1.1 入口与监管模型

`Dockerfile` 关键变更：

- `ENTRYPOINT` 从 `/usr/bin/tini` 切到 `/init`（s6-overlay v3）。
- 主程序 `hermes` 通过 `docker/main-wrapper.sh:1-30`（30 行）作为 **CMD** 运行（不是 s6-rc 服务），保留 TTY 透传与 `--tui` 兼容。原因（commit message）："cont-init.d scripts receive no arguments → CMD args 不可见 stage2-hook；`/run/s6/basedir/bin/halt` 后退出码丢失（容器固定 143）" — 因此主进程仍走 CMD，但 dashboard 与 per-profile gateway 由 s6 监管。
- `docker/stage2-hook.sh:1-142` 取代旧 `entrypoint.sh` 内嵌的 gosu 调用，做：UID/GID 对齐 → `chown` → 服务目录骨架预创建 → `s6-setuidgid` 切到 `hermes` 用户（UID/GID `10000`，硬编码于 `hermes_cli/service_manager.py:347-348`）。
- `gosu` 二进制从镜像中彻底移除（`4b4c36cb6 feat(docker): remove gosu from bundled image`）—— s6 自带 `s6-setuidgid` 完成权限降级。

### 1.2 新增 `ServiceManager` Protocol + 4 后端

`hermes_cli/service_manager.py:54-309`（886 行总计，新增 530+ 行）：

## 4. CLI 冷启动 -63% — Bitwarden 磁盘 L2 缓存（perf）

`0219b0408 perf(cli): cut hermes startup 63% — flip head-to-head vs codex (#31968)`：

- 旧逻辑：每次 `hermes` 调用都跑 `bws secret list`（Bitwarden Secrets Manager），耗时 ~380ms，内存 cache 不跨进程。
- 新逻辑：磁盘 L2 cache `<hermes_home>/cache/bws_cache.json`，mode `0o600`，**只存** SHA-256 fingerprint 前缀（access token 不落盘）。同 TTL，miss 时读盘，bws 成功时写盘，key mismatch / 损坏 / 过期时忽略。

测得数据（commit message）：

| 指标 | Before | After |
|-----|--------|-------|
| `load_hermes_dotenv()` 冷启 | 372ms | 20ms（warm 盘 cache hit） |
| `hermes --version` cold→warm | 666ms | ~295ms |
| single-turn median (11 tasks × 3 trials) | 2.96s | 2.31s（-0.65s） |
| 5-turn multi-turn | 9.40s | 8.95s（-0.45s，约 0.3s/turn） |

定位：`hermes_cli/dotenv.py`（commit message 描述，未直接看）。

## 7. CredentialPool — 周配额轮换正确性（fix）

`4117fc364 fix(credential-pool): correct pool rotation when weekly usage limit is reached`：

- **根因**：当 key #1 标记 `exhausted` 后，`_get_cached_client` / `resolve_api_key_provider_credentials` 因环境变量偏好（`OPENAI_API_KEY` 等）**仍然返回 key #1 的 cached client**，retry 用 key #1 又一次撞上限。
- **修复**：retry 路径 `peek()` pool 拿活跃 entry 的 key，作为 `explicit_api_key` 强制传入；不再依赖 env-var fallback。
- 附带修复：
  - `mark_exhausted_and_rotate(api_key_hint=...)` —— 并发 CLI + gateway 场景下精确定位要标记的 entry（防止 CLI 标记 key A 而 gateway 还在 cached B 上）。

## 8. Gateway — 在飞子 Agent 抗 busy-mode 中断（fix #30170）

`99d62f6ba fix(gateway): protect in-flight subagents from busy-mode interrupts (#30170)`：
