## 2. Kanban DB 抗污染（fix + feat + infographic）

**PR #30858 + #30862 + #30949 + #30952**，commit `39fe4ec` + `c4b8f5e` + `ad11327` + `cae7537`：v0.14.0 用户社区出现的"kanban.db 神秘损坏 → 自动重建覆盖旧任务"事故的根因 + 防御。

### 2.1 拒绝在损坏 DB 上自动初始化（`39fe4ec`）

`hermes_cli/kanban_db.py:1010-1132`，新增三层结构：

```
connect(path)                # kanban_db.py:1135
  └─ _guard_existing_db_is_healthy(path)   # :1077-1132
       ├─ resolve() + size>0 检查
       ├─ sqlite3.connect(...) PRAGMA integrity_check
       ├─ ok        → 加入 _INITIALIZED_PATHS 缓存
       ├─ corrupt   → _backup_corrupt_db(path) + raise KanbanDbCorruptError
       └─ Locked    → 让 sqlite3.OperationalError 透传（不当作 corrupt）
```

- **失败闭合**：任何 `PRAGMA integrity_check` 非 `ok` 都 raise `KanbanDbCorruptError`（`kanban_db.py:1010-1026`），调用者不能"无声重建"。
- **保留证据**：损坏 DB 拷到 `<db>.corrupt.<YYYYMMDD_HHMMSS>.bak`，WAL/SHM sidecar 一并备份（`kanban_db.py:1029-1074`）。
- **缓存友好**：成功探针的路径写入 `_INITIALIZED_PATHS`，同进程后续 `connect()` 不重跑 integrity（`kanban_db.py:1113`）。
- **锁争用不误判**：`sqlite3.OperationalError` 直接 `raise`，不当作损坏，避免对正在 WAL checkpoint 的 DB 误备份（`kanban_db.py:1124-1126`）。

### 2.2 备份路径硬化对抗 CodeQL path-injection（`c4b8f5e`）

`hermes_cli/kanban_db.py:1029-1074`：所有写发生**在 `path.resolve().parent` 之内**，且每次构造 `candidate = parent / basename_only` 后断言 `candidate.parent != parent → return None`（`kanban_db.py:1051-1052, 1057-1058, 1069-1070`）。WAL/SHM sidecar 同样走 resolved-parent 路径，`..` 段在任何 I/O 之前被 collapsed。

功能等价（同样 raise `KanbanDbCorruptError`、同样 `.corrupt.<ts>.bak` 命名），仅给静态分析提供 containment 证据。163/163 kanban 测试通过。

### 2.3 Scratch workspace 一次性提示（`ad11327`）

PR **#30949**：用户社区报告"progress files vanished, no warning anywhere"。

`hermes_cli/kanban_db.py:3109-3181` —— 首次在该 install 创建 scratch workspace 时：

1. 一次性 warning log："scratch workspaces are ephemeral — they're deleted when the task..."（`kanban_db.py:3117-3121`）
2. 在 task 上 append `tip_scratch_workspace` 事件 —— Dashboard 等读 `task_events` 表的消费者可见（`kanban_db.py:3174`）
3. Touch sentinel `~/.hermes/kanban/.scratch_tip_shown`（`kanban_db.py:3141-3181`），整 install 静默后续创建

行为不变 —— scratch **仍然**是 ephemeral by design，只是把设计可见性补齐。docs（en + ko）相应在 scratch bullet 加 "Deleted when the task completes"，worktree/dir 标 "Preserved on completion"。

## 3. Memory.md / USER.md 外部漂移防护（fix #26045）

PR **#30877**，commit **`6855d17`**，`tools/memory_tool.py` +116 / `tests/tools/test_memory_tool.py` +125。

### 3.1 重现场景

生产，2026-05-14：两个并发 session 跑同一个 agent。
- Session A 用 patch tool 直接追加 ~8KB 结构化内容（Vendor Master / Standing Orders / Pin Board），**没有 § 分隔符** —— 因为没走 memory tool。
- Session B 启动时只看到 1 个 entry（~331 chars，in-memory cache 是旧的）。
- Session B 调 `memory(action=replace, name=<old>, content=<333 字节>)`。
- Memory tool 的 `_read_file` 把整 8KB 内容解析为**一个** entry（没有 § splits），`replace` 把它截到 333 字节 —— **~8KB 结构化内容静默被销毁**。

原子 rename 写路径本身没问题，问题在 implicit 契约：tool 假设 `MEMORY.md` 是它自己用 § 分隔写的小条目列表，但 v0.13 安装 runbook 自己就用 `cat >> MEMORY.md`，patch tool / operator 也直接编辑。

### 3.2 `_detect_external_drift` 双信号（`tools/memory_tool.py:482-530`）

```python
def _detect_external_drift(self, target):
    raw = read(path)
    parsed = self._parse(raw)
    roundtrip = self._serialize(parsed)
    max_entry_len = max(len(e.content) for e in parsed)
    drift_detected = (raw.strip() != roundtrip) or (max_entry_len > char_limit)
    if not drift_detected: return None
    # 备份 → 拒绝 mutation
    bak_path = path.with_suffix(path.suffix + f".bak.{ts}")
    ...
```

信号 1：**roundtrip 不等价** —— 重新 serialize 后字节级不同，说明分隔符 / 编码异常（`memory_tool.py:522`）。
信号 2：**最大单 entry 超过整 store 限额**（memory=2200, user=1375 chars）—— tool 永远不会写出超过整 store 限额的单条 entry，所以这说明外部 writer 把自由文本塞进了 tool 视为单 entry 的位置（`memory_tool.py:522`）。

### 3.3 检测触发后

- `_reload_target` 写 `.bak.<ts>` 快照（`memory_tool.py:219-231, 530`）
- add / replace / remove **拒绝 flush**（`memory_tool.py:277-282, 329-331, 381-383`）—— 原文件保持不变
- 错误字典携带 `.bak` 路径 + remediation 字串："integrate missing entries via memory(add=...) one at a time, then rewrite the file clean"（`memory_tool.py:108-130 _drift_error`）—— 模型可直接行动，不必上升给 operator
- 同样保护 USER.md（不只是 MEMORY.md）

测试 137 → 144（+7）。

## 5. TLS FD 回收竞态最终修复（fix #29507）

三连击 commit **`e2a7d73`** + **`30c22f1`** + **`5b6f0b6`**：`force_close_tcp_sockets` 仅 shutdown 不 close、`client.close()` 跨线程时改为 abort+defer、回归测试。

### 5.1 故障链

生产事故：`kanban.db` 文件在没有 schema 改动的情况下被 24 字节 TLS application-data record 改写。

根因（`agent/agent_runtime_helpers.py` commit 信息）：

```
helper 调 socket.shutdown(SHUT_RDWR) + socket.close()
                                       ↑ 后者从 stranger thread 调用
release 该 FD 整数到内核
内核把同一个整数 recycle 给下一个 open() → kanban dispatcher 的 kanban.db
但 httpx worker 还在 SSL BIO 缓存里持有同一个 FD 整数
worker 延迟的 TLS flush 写一个 24-byte 应用数据 record
                                  ↓
            写到了 kanban.db 的 SQLite header 上
```

### 5.2 Layer-1：`force_close_tcp_sockets` 缩窄到 shutdown-only（`e2a7d73`）

`agent/agent_runtime_helpers.py` +55 −18：移除 `socket.close()`，只保留 `shutdown(SHUT_RDWR)`（任意线程安全 —— 只发 FIN + 中断 `recv`/`send`，**不释放 FD**）。owning httpx worker 自己 unwind 时会通过同一个 Python `socket.socket` 对象 close，那个对象在 close(2) 之前把 `_fd` 原子置 -1，没有 FD aliasing 窗口。

日志字段 `tcp_force_closed=N` 保留语义（now counts shutdowns），dashboard / 日志解析器无需改动。

### 5.3 Layer-2：跨线程 client.close() 改为 abort + defer（`30c22f1`）

`agent/chat_completion_helpers.py` +65 / `run_agent.py` +33：即便 socket helper 只 shutdown，后续 `client.close()` 走 httpx pool 时还是会 close socket，跨线程仍有 FD-recycle 窗口。

机制：

```python
def _set_request_client(client):
    request_client_holder["thread_ident"] = threading.get_ident()
    request_client_holder["client"] = client

def _close_request_client_once(...):
    stranger_thread = (
        request_client_holder.get("thread_ident")
        != threading.get_ident()
    )
    if stranger_thread:
        agent._abort_request_openai_client(request_client, reason=reason)
        # shutdown(SHUT_RDWR) the pool sockets + 写 deferred_close 标记
        # holder 保持 populated → owning worker 的 finally 会做真正 close
    else:
        agent._close_request_openai_client(request_client)
        # 正常 close
```

`agent/chat_completion_helpers.py:97-141`（非流式 `interruptible_api_call`）和 `:1312-1345`（流式版本）对称应用。

日志字段 `tcp_force_closed=N` 维持原 shape，新增 `deferred_close=stranger_thread` 区分两种 close 通道。

### 5.4 Layer-3：回归测试 (`5b6f0b6`)

`tests/run_agent/test_create_openai_client_reuse.py` +12 行：pin shutdown-only + thread-aware close contract。

三层防御独立，任一回滚不会立刻重开 advisory。

## 6. GHSA-5qr3-c538-wm9j 第二段：Plugin RCE 双保险（fix）

PR **#29156**，commit `09f85f2` + `8bf9922` + `da636e9` + `6a2df9f`：v0.13.0 安全 wave 第一段已修了 truthy env-gate；第二段堵 `_mount_plugin_api_routes` 任意文件 import。

### 6.1 攻击面

`hermes_cli/web_server.py:_mount_plugin_api_routes` 把每个 dashboard plugin 的 manifest `api` 字段以 `importlib.util.spec_from_file_location` 当 Python 模块**导入** —— 设计上就是任意代码执行。两个原本无害的原语让它变可利用：

1. **绝对路径吞噬目录**：`Path('safe/dashboard') / '/tmp/evil.py'` resolve 成 `/tmp/evil.py`，manifest 一行 `{"api": "/tmp/payload.py"}` 就能把 importer 重定向到磁盘任意文件。
2. **`..` 遍历爬出 dashboard 目录**：`Path('plugins/safe/dashboard') / '../../../tmp/evil.py'` 在 `resolve()` 之后落在 `/tmp/evil.py`。静态资源 handler（`serve_plugin_asset`）已经用 `is_relative_to` 防住，api-mount 路径没防。

### 6.2 三层修复

`hermes_cli/web_server.py` +88（commit `8bf9922`）：

1. **`_safe_plugin_api_relpath`** validator 在**发现期**跑（`web_server.py:4050`）：拒绝绝对路径、`..` 遍历、空 / 非字符串值、resolve 出 `dashboard/` 目录的路径，警告点名 plugin。`has_api` follow sanitized 值，前端不渲染假的 "Backend API" badge。
2. `_mount_plugin_api_routes` 在 import **之前再次** 对 resolved path 验活体文件系统（line 4547 `api_file_name = plugin.get("_api_file")`）—— 防 `_dir` 被 post-cache 篡改 / 未来 caller 绕过 discovery validator。
3. **Project plugins (`source == "project"`) 直接拒绝** backend import —— `./.hermes/plugins/` 跟着 CWD 走，"用户打开恶意 repo" 的威胁模型把它当攻击者可控。Project plugins 仍可用静态 JS/CSS 扩展 UI，但 Python `api` 不再自动 import。

加上前一 commit (`09f85f2`) 的 **truthy env-gate fix**（`HERMES_ENABLE_PROJECT_PLUGINS` 现在按 truthy 解释，不是只看 != "0"），advisory chain 在**两个独立 choke point** 失败。`tests/.../test_plugin_rce_regression.py` 钉死回归（`da636e9`）。
