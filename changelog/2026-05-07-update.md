## Multi-agent Kanban

`tools/kanban_tools.py` (1139 行) + `hermes_cli/kanban.py` (2252 行) + `hermes_cli/kanban_db.py` (4839 行) + `hermes_cli/kanban_diagnostics.py` + `hermes_cli/kanban_specify.py` —— 一个**全新的多 agent 协同子系统**。

### 设计要点（源码验证）

**SQLite + WAL + BEGIN IMMEDIATE + CAS**（`kanban_db.py:60-78`）：
- WAL 模式 + `BEGIN IMMEDIATE` 串行化 writer
- `tasks.status` / `tasks.claim_lock` 用 **compare-and-swap** 更新
- 最多一个 claimer 能赢任一任务，失败方观察到 0 affected rows 后走开 —— 零重试循环、零分布式锁

**Schema**（最小化）：
- 表：`tasks` / `task_links` / `task_comments` / `task_events`
- 字段 `workspace_kind`：`scratch` / `worktree` / `dir` —— 解耦 worktree 与协调，让 research / ops / digital-twin 工作流也能跑

**Claim TTL**：15 分钟。Worker 需要周期性 `heartbeat_claim(task_id)`，否则下次 dispatcher tick 会**回收**任务（防 zombie worker）。

**多 board**（`kanban_db.py:9-50`）：
- 默认 board 叫 `default`，路径 `<root>/kanban.db`（向后兼容旧版无 boards 用户）
- 额外 board 放 `<root>/kanban/boards/<slug>/`，独立 `kanban.db` / `workspaces/` / `logs/`
- Worker 只看到自己 task 所在的 board，不能枚举其它 board

**Board 解析优先级**（高到低）：
1. `connect(board=...)` 显式参数
2. `HERMES_KANBAN_BOARD` env
3. `HERMES_KANBAN_DB` env（直接钉死 DB 文件路径，legacy）
4. `<root>/kanban/current` 单行文件（`hermes kanban boards switch <slug>` 写入）
5. 默认 `default`

**Worker 子进程注入**（dispatcher 主动把以下注入 env）：
- `HERMES_KANBAN_DB` / `HERMES_KANBAN_WORKSPACES_ROOT` / `HERMES_KANBAN_BOARD`
- 这样在不寻常的 symlink / Docker 部署下 worker 也能 converge 到 dispatcher 用的 DB

**Worker tools 是 schema-gated**（`kanban_tools.py:1-30`）：
- 只在 `HERMES_KANBAN_TASK` env 存在时**才进 model schema**
- 普通 `hermes chat` 看不到 kanban 工具
- 三个走 CLI / dashboard / `/kanban` 斜杠 → 完全 bypass agent

详见 [[multi-agent-architecture]]（待新建 kanban 子页面）。

## `/goal` — Ralph Loop

`hermes_cli/goals.py`（593 行，已源码验证）—— **持久化跨轮目标**。

### 工作流

1. 用户 `/goal <text>` 设置目标，写入 SessionDB 的 `state_meta` 表，key = `goal:<session_id>`
2. 每轮结束后，一个小 judge 调用（aux 模型）问 "这个 goal 在 assistant 上一句回复里被满足了吗？"
3. **不满足 → 继续**：自动给 session 追加一条 normal user message（continuation prompt），重新喂给 `run_conversation`
4. **满足 / turn budget 耗尽 / 用户 pause / 用户发新消息 → 停**：用户消息优先级最高，会**抢占** continuation 并暂停 goal loop（但仍 re-judge —— 用户消息恰好满足 goal 的话 judge 会说 done）

### 不变量

- continuation prompt 是**普通用户消息**，不动 system prompt，不换 toolset —— **prompt cache 完全不失效**
- Judge 失败 **fail-OPEN（continue）**：broken judge 不能 wedge 进度，turn budget 是 backstop
- Judge **解析失败累计 3 次**（默认 `DEFAULT_MAX_CONSECUTIVE_PARSE_FAILURES = 3`）后自动 pause，提示用户检查 `goal_judge` 配置
- API / transport 错误**不计入** 3 次（那是 transient）
- `/resume` 能恢复 goal —— 状态在 SessionDB 里
- 默认 `DEFAULT_MAX_TURNS = 20`；`DEFAULT_JUDGE_TIMEOUT = 30s`；`_JUDGE_RESPONSE_SNIPPET_CHARS = 4000`

### Judge 提示词（源码 `goals.py:80-110`）

```
A goal is DONE only when:
- The response explicitly confirms the goal was completed, OR
- The response clearly shows the final deliverable was produced, OR
- The response explains the goal is unachievable / blocked / needs
  user input (treat this as DONE with reason describing the block).
```

Judge 用 JSON-only 单行回复，便于解析。

### Post-write Delta Lint（`write_file` / `patch`）

`tools/file_operations.py:_check_lint_delta()`（`file_operations.py:1089`）：

写入完成后自动 lint。**仅报本次新增的错**（pre/post 差集），pre-existing 问题不会让 agent 走偏。in-process linter（Python AST / JSON / YAML / TOML 等）零开销；shell linter 跳过 pre-state 抓取（避免 re-write-read 的循环），接受 naive "all errors" 报告。

源码 `file_operations.py:830-840`：

> After the write, runs a post-first / pre-lazy lint check via `_check_lint_delta()`. If the new content is clean, the lint call is O(one parse). If the new content has errors, the pre-write content is linted too and only errors newly introduced by this write are surfaced — pre-existing problems are filtered out so the agent isn't distracted chasing them.

### Sessions Survive Restarts

`gateway/run.py:3124-3192` + `gateway/session.py:992-1095` 实现**重启后自动恢复**：

- gateway 中途崩 / `/update` 重启 / 源码文件变更触发自重启

## Checkpoints v2

`tools/checkpoint_manager.py` —— **单一共享 shadow git store**（重写）。

### 旧 vs 新

| 旧（v1） | 新（v2） |
|---------|---------|
| 每个 working dir 一个 shadow repo | 单一共享 store：`~/.hermes/checkpoints/store/` |
| 同一 repo 的 12 个 worktree = 12 份重复 objects | git object DB 自动跨 project 去重 |
| 一个重 user ~500 MB shadow 数据 | 加新 worktree 几乎零成本 |

**Storage layout**：

```
~/.hermes/checkpoints/
    store/
        HEAD, config, objects/      — git internals（共享）
        refs/hermes/<hash16>        — per-project branch tip
        indexes/<hash16>            — per-project git index
        projects/<hash16>.json      — {workdir, created_at, last_touch}
        info/exclude                — 默认 excludes
    .last_prune                     — auto-prune 幂等标记
    legacy-<timestamp>/             — 旧 per-project shadow（自动迁移）
```

**Auto-maintenance**：`prune_checkpoints` 删 orphan refs（workdir 消失）+ stale refs（last_touch 超过 `retention_days`）+ `git gc --prune=now`；额外 size-cap pass 按"最旧优先"删到 `max_total_size_mb` 之下。

**隔离**：用 `GIT_DIR` + `GIT_WORK_TREE` + `GIT_INDEX_FILE`，**不污染**用户项目的 `.git/`。

详见 [[worktree-isolation]]。
