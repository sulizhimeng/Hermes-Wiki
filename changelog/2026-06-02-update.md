## 4. Dashboard auth refresh-token 轮转

**Commit**：`c10ccaaf feat(dashboard-auth): rotate dashboard sessions via refresh token (#37247)` — 4 文件 +428 / -55。

之前如果浏览器在 `Max-Age` 后驱逐 access-token cookie（refresh-token 还在），网关会**硬 401 触发完整重登**。现在能用 refresh token 静默续期。

### 4.1 Provider 端（`plugins/dashboard_auth/nous/__init__.py`）

- **新方法**：`:244-302 refresh_session(refresh_token)` — POST `grant_type=refresh_token` 到 Portal `/token`
- **请求体**：`:269-291` refresh_token 在 body（Portal schema 要求）**和** `x-nous-refresh-token` header 同时发送（log redaction）
- **空 RT fast-fail**：`:263-266` 无网络调用直接 `RefreshExpiredError`
- **错误映射**：`:298-302` `400` → `RefreshExpiredError`（过期/吊销/重放检测），网络错 → `ProviderError`
- **共享处理器**：`:304-326 _token_response_to_session()` — 参数化 exception 类型（auth-code 用 `InvalidCodeError`，refresh 用 `RefreshExpiredError`）
- **初次登录捕获 RT**：`:207-242 complete_login()` — 前向兼容：若 Portal 不返回 RT 则存为空串

### 4.2 Middleware 端（`hermes_cli/dashboard_auth/middleware.py`）

- **关键修复**：**AT 缺席 + RT 存在** 现在走 refresh 路径，**不再硬 401 强制重登**
- 实证（commit body 摘自 `git show c10ccaaf` message）：
  - 之前：`_attempt_refresh` 只在 AT **存在但失效**时才跑（这在浏览器里几乎不会发生）
  - 之后：只有**两个 cookie 都缺**时才硬 bounce
- **核心调用**：`:236 _attempt_refresh(request, refresh_token=_rt)`、`:255 refresh_token=new_session.refresh_token` 写回新 cookie
- **新函数**：`:302 _attempt_refresh(request, *, refresh_token)`

### 4.3 测试

- `tests/hermes_cli/test_dashboard_auth_401_reauth.py` **+89 行**（3 个回归 case：AT 驱逐 + RT 存在 / 双 cookie 缺 / RT 只 + 死 RT）
