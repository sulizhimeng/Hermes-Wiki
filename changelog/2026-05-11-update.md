## 顶层亮点

### 1. Cross-session 1h prefix cache for Claude（#23828）

`agent/prompt_caching.py` 的 `prefix_and_2` 策略**正式落地**到 Anthropic / OpenRouter / Nous Portal 的 Claude 模型：

- **首轮新 session 的输入成本砍 ~85–90%**（前提是同一 user config 1h 内有过任一轮）
- 4 个 cache_control breakpoint 拆**两个 TTL 层级**：
  - `tools[-1]` (1h) + 稳定 system prefix block[0] (1h) —— 跨 session 共享
  - 末尾 2 条非 system message (5m) —— session 内滚动
- 之前的 `system_and_3` 是单 TTL（5m 或 1h），4 breakpoint 全在末尾窗口
- 调度顺序由 Anthropic prefix-cache 决定：`tools → system → messages`
- volatile suffix（memory / USER profile / 时间戳 / session_id）扔到第二个 block，**不污染**第一个 block 的 byte-stable prefix

源码（`agent/prompt_caching.py:10-22`）：

> `prefix_and_2` (Claude on Anthropic / OpenRouter / Nous Portal): 4 breakpoints split across two TTL tiers — tools[-1] (1h) + stable system prefix (1h) + last 2 non-system messages (5m). The long-lived prefix is byte-stable across sessions for a given user config, so every fresh session reads the cached system+tools instead of re-paying for them. Within-session rolling window shrinks from 3 messages to 2 to free the breakpoint budget.

详见 [[prompt-caching-optimization]]。

### 2. `HERMES_SESSION_ID` 通过 ContextVar 暴露给 agent tools（#23847）

`gateway/session_context.py` 加 `_SESSION_ID` ContextVar + `_VAR_MAP` 入口。

**为什么用 ContextVar 不用 os.environ**：gateway 一个进程里可能并发多个 session，纯 `os.environ` 会**跨 session 串扰**。ContextVar 在 asyncio task 之间隔离干净。CLI 模式没有并发问题，仍写 `os.environ` 作为 fallback。

**Touchpoints**（源码验证）：
- `gateway/session_context.py` —— 注册 `_SESSION_ID` 进 ContextVar 表
- `agent/run_agent.py` —— init 时 + 上下文压缩 session 轮换时同时写 ContextVar + os.environ
- `tools/environments/local.py:_make_run_env()` —— 把 ContextVars **桥接进子进程 env**（ContextVar 不会自动跨进程）
- 与已有的 `tools/env_passthrough.py` / `tools/credential_files.py` 套同一套 ContextVar pattern

**消费方**：
- `agent/skill_preprocessing.py` 的 `${HERMES_SESSION_ID}` SKILL.md 模板替换
- `execute_code` / `terminal` 工具的子进程 env（`HERMES_*` 前缀已 passthrough 列表）

## Sudo 安全加固（多 commit 集中点）

`fix(approval): catch sudo with stdin/askpass/shell privilege flags` + `fix(terminal): block sudo -S password guessing when SUDO_PASSWORD is not set` —— sudo 命令审批层的窗口收紧：

- 现在能识别 `sudo -S`（stdin 喂密码）/ `sudo --askpass` / shell 提权 flag
- `SUDO_PASSWORD` 未设时拒绝 `sudo -S` 路径，避免 agent 用 `echo "" | sudo -S ...` 蒙混密码 prompt
- AUTHOR_MAP 收录 salvage 贡献者

详见 [[security-defense-system]]。
