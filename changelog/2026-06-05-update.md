2. 用 PKCE (S256) 构建 `/authorize` URL
3. 在发现的 `token_endpoint` 交换 code → tokens
4. **验证 ID token**（RS256/ES256）via 发现的 `jwks_uri`，`iss`/`aud` 钉死 issuer/client_id，映射 OIDC 标准 claim（`sub`/`email`/`name`）到 `Session`

**Public PKCE clients only**：confidential client（带 client_secret）暂不支持，有 `# TODO(confidential-client)` 缝在 `complete_login` / `refresh_session`。自托管者注册 CLI/SPA 客户端几乎都是 public + PKCE，这条路径表面更小。

## 7. Discord — 语音频道软件混音器

**Commit**：`8a9ded5b2 feat(discord): voice-channel mixer — ambient idle bed + verbal acks that overlap TTS`（6 文件 +952/-1）。

### 7.1 问题与设计

discord.py（Rapptz）一个 voice connection 一条 opus 流：`VoiceClient.play()` 接单个 `discord.AudioSource`，调用时如果已经 playing 会抛 `ClientException`。

**新建** `plugins/platforms/discord/voice_mixer.py 378 行`：

- `:148 class VoiceMixer` —— 自己就是 `discord.AudioSource`，discord.py 每 20ms 调 `read()`。内部把任意数量 child source 的 20ms PCM 帧加和、clamp 到 int16、返回一个混合帧。discord.py 完全感知不到下面合了几路。
- 一条 voice connection 一次能提供：
  - 始终在跑的低音量 **ambient / idle bed**（"thinking" 底色）
  - **speech 通道**（TTS 回复、口头 ack）**覆盖** ambient bed，发声时自动 **duck** ambient gain，结束恢复 —— Grok-voice 那种平滑感
- 设计要点（源码注释 `:36-50`）：
  - mixer 在 guild 加入时安装 **一次** (`vc.play(mixer)`)，bot 离开前不停 —— 子流来来去去，mixer 永不 stop，所以 ack 与最终回复之间没有 `is_playing()` race
  - 帧格式 Discord 原生：48kHz / 2ch / int16 LE / 20ms（3840 字节 == `discord.opus.Encoder.FRAME_SIZE`）
  - 混音是 **单次向量 int32 加 + clip**（numpy，已是 `voice` extra 依赖），CPU 开销可忽略
  - `read()` 在 discord.py audio sender 线程，children 增删在 asyncio loop 线程 —— shared state 用 `threading.Lock` 守
- mixer 仅产 **outbound** 流：`VoiceReceiver` 只解 inbound SSRC，所以 mixer 输出不会回声进 transcription

## 15. TUI — cgroup-aware V8 heap cap

**Commit**：`2f523a469 fix(tui): cgroup-aware V8 heap cap so memory-limited containers stop dying silently`。

`ui-tui/src/lib/memoryMonitor.ts:20-50` 解释：之前阈值是硬编码 `2.5 GB`，在 8GB heap ceiling（`--max-old-space-size=8192`）下相当于 ~31% 就触发 critical exit + 静默关 gateway stdin。

新逻辑（HEAD 实证）：

```typescript
const GB = 1024 ** 3
const MB = 1024 ** 2

function resolveThresholds(criticalBytes?: number, highBytes?: number) {
  let limit = 0
  try {
    limit = getHeapStatistics().heap_size_limit || 0
  } catch {
    limit = 0
  }

  // Fall back to the historical 8GB ceiling if V8 doesn't report one.
  const ceiling = limit > 0 ? limit : 8 * GB
  const critical = criticalBytes ?? Math.max(2 * GB, Math.round(ceiling * 0.88))
  const high = highBytes ?? Math.max(1 * GB, Math.min(critical - 256 * MB, Math.round(ceiling * 0.7)))

  return { critical, high }
}
```

—— **critical ~88%**, **high ~70%** of real V8 ceiling，配 sane floor/ceiling clamp 防过小 `--max-old-space-size` 把阈值压到 warn watermark 下。

`ui-tui/src/entry.tsx:1 #!/usr/bin/env -S node --max-old-space-size=8192 --expose-gc` shebang 把 ceiling 钉到 8GB。

**影响**：在 1GB 容器 / 受 cgroup 限制的 Docker 里跑 TUI 不再因为没到 2.5GB 就静默 OOM kill。
